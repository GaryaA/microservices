# Summary: Section 1 - Introduction to Microservices Architecture

## Обзор секции

В первой секции курса изучались основы микросервисной архитектуры. Это вводная секция, которая закладывает фундаментальные знания о микросервисах, их преимуществах, недостатках и случаях использования.

## Что было изучено

### 1. Монолитная архитектура
- Традиционный подход к разработке приложений
- Все компоненты в одном приложении
- Преимущества:
  - Простота разработки и развертывания
  - Простая отладка
  - Транзакционная целостность
- Недостатки:
  - Сложность масштабирования
  - Тесная связанность компонентов
  - Медленное развертывание
  - Технологические ограничения

### 2. Микросервисная архитектура
- Разделение приложения на небольшие независимые сервисы
- Каждый сервис отвечает за определенную бизнес-функцию
- Коммуникация через хорошо определенные API
- Независимое развертывание и масштабирование

### 3. Преимущества микросервисов
- **Независимое развертывание** - обновление одного сервиса не влияет на другие
- **Технологическое разнообразие** - разные технологии для разных сервисов
- **Масштабирование** - масштабирование только нужных сервисов
- **Отказоустойчивость** - изоляция сбоев
- **Командная автономия** - разные команды работают над разными сервисами
- **Повторное использование** - сервисы могут использоваться разными клиентами

### 4. Недостатки микросервисов
- **Сложность** - больше компонентов для управления
- **Распределенные транзакции** - сложность обеспечения консистентности
- **Сетевая задержка** - коммуникация через сеть
- **Дублирование данных** - возможное дублирование между сервисами
- **Тестирование** - сложность интеграционного тестирования
- **Мониторинг** - необходимость отслеживания множества сервисов

### 5. Когда использовать микросервисы
- Большие и сложные приложения
- Необходимость независимого масштабирования компонентов
- Разные команды разработки
- Различные требования к технологиям
- Высокие требования к доступности

### 6. Когда НЕ использовать микросервисы
- Небольшие приложения
- Простые бизнес-процессы
- Небольшие команды
- Отсутствие опыта в распределенных системах
- Строгие требования к транзакциям

### 7. Ключевые принципы
- **Single Responsibility** - каждый сервис отвечает за одну функцию
- **Independence** - сервисы независимы друг от друга
- **Decentralization** - децентрализованное управление данными
- **Failure Isolation** - изоляция сбоев
- **API First** - хорошо определенные API

### 8. Вызовы микросервисов
- Service Discovery - как сервисы находят друг друга
- Configuration Management - управление конфигурациями
- API Gateway - единая точка входа
- Distributed Tracing - отслеживание запросов
- Security - безопасность в распределенной системе
- Data Management - управление данными

## Ключевые концепции

### Domain-Driven Design (DDD)
- Определение границ сервисов на основе доменных моделей
- Bounded Context - границы контекста
- Event Storming - техника для определения границ

### API Design
- RESTful API
- GraphQL (опционально)
- gRPC для внутренней коммуникации
- Асинхронная коммуникация через messaging

### Data Management
- Database per Service - каждый сервис имеет свою БД
- Saga Pattern - для распределенных транзакций
- Event Sourcing - для аудита и восстановления состояния
- CQRS - разделение чтения и записи
